## 1. 使用技巧

### 1.1 多实践

用好 AI 是需要技巧的，而技巧来自实践。多用，才能真正摸清它的边界和潜力。有了想法，不必等到想清楚再动手。先做起来，问题自然会浮现。

### 1.2 数据来源

尽可能从多个可靠的渠道获取数据，交叉比对，避免单一来源带来的偏差。

### 1.3 规则适度

给 AI 加更多规则并不总是好事。对于确定性强的场景，规则能有效减少幻觉；但规则过多，反而会束缚 AI 的发挥空间。

### 1.4 边界

不要止步于 AI 给出的第一个方案，持续追问、多角度验证，甚至启动多智能体协作，才能真正探到边界。

### 1.5 提示词

提示词可以写得简单，也可以结构化地组织。不确定怎么写时，可以让 AI 帮你优化，或者直接让它调用相应的智能体来完成任务。最重要的是，动手之前心里要有预期——你想要什么结果，AI 才能更准确地给到你。写提示词是一项可以持续打磨的技巧，用得越多，越知道怎么问。

### 1.6 提问

学会发现问题，学会提问，学会追问。

## 2. 安全

### 2.1 API Key 泄露

有程序员不小心把 LLM 的 API Key 提交到了公开的 GitHub 仓库，被自动扫描工具发现后迅速遭到滥用，账单暴涨，损失惨重。

这类事故并不罕见。Key 一旦泄露，攻击者便可用你的额度调用模型，费用全算在你头上。防范方法并不复杂：将 Key 存入环境变量，不要硬编码进代码，提交前用 `.gitignore` 排除 `.env` 文件。

### 2.2 软件供应链攻击

Apifox 曾被曝出远程代码注入漏洞，攻击者可借此窃取用户本地的敏感数据，包括 SSH 私钥、环境变量等。

开发工具本身成为攻击入口，这类供应链攻击的危险在于隐蔽性强——你信任的工具在后台做了你不知道的事。防范上，SSH 私钥建议用密码短语加密，不同服务使用独立的密钥对，敏感凭证不要集中存放在容易被读取的位置。

### 2.3 敏感数据与 AI 工具

使用 AI 编程助手时，不要将 API Key 直接粘贴进对话或代码片段。应通过环境变量或系统 Keychain 管理凭证，让工具按需读取，而非明文暴露。

不要将高度敏感的数据（如公司财务数据、用户隐私）直接提交给公共 AI 服务分析。如果业务确有需要，应通过官方 API 或自建私有模型处理，而非使用第三方聊天界面。

## 3. 人与 AI

AI 是不可信的，值得信任的是使用它的人。

AI 放大个人的能力，但对人的判断力要求也随之提高。工具越强，使用者的决策质量就越关键。

AI 生成的内容，你有没有认真思考过？有没有用批判思维审视它？有没有用不同的工具交叉验证？这些习惯，决定了 AI 究竟是在帮你，还是在误导你。

AI 正在改变人的工作方式。值得思考的是：能从 AI 身上学到什么，哪些工作可以交给它来做。AI 放大执行力，但真正驾驭它的，还是有想法的人。
